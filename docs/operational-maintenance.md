# Operational Maintenance — Cleanup Semantics & Runtime Retention

Clarifies **what each verb does**, **what survives**, and **how to manage
disk growth** without changing runtime code.

---

## 1. Scenario stop vs lab cleanup

| Action | Command | What it does | CALDERA / libvirt |
| --- | --- | --- | --- |
| **Scenario stop** | `aella_cli lab scenario stop` | Finishes or cancels the **active CALDERA operation**; updates local scenario state / JSONL | **Does not** destroy VMs or revert disks |
| **Lab cleanup** | `aella_cli lab cleanup` | Engine **`cleanup all`**: **stop + destroy** every VM in `lab-vms.json` | **Undefines** domains, removes **runtime** qcow2 files |

Use **stop** after each scenario for server hygiene; use **cleanup** only
when you intend a **full lab teardown**.

---

## 2. VM stop vs destroy vs snapshot revert

| Verb | Command | Guest disk | Domain definition |
| --- | --- | --- | --- |
| **Stop** | `aella_cli lab stop <vm>` or `aella_cli lab stop all` | Retained | Retained |
| **Destroy** | `aella_cli lab destroy <vm>` or `aella_cli lab destroy all` | **Runtime** qcow2 removed | Undefined |
| **Snapshot revert** | `aella_cli lab snapshot revert <name>` | Rewinds to named snapshot chain | Retained |

**Destroy** does **not** remove cached **base** images under
`${XDR_IMAGES_DIR}`; it removes the mutable **runtime** copy pattern
documented in `docs/skills/kvm-runtime-skill.md`.

---

## 3. What is preserved vs intentionally persistent

**Preserved across `lab stop`:**

- `${XDR_BASE}/images/` caches and golden qcow2 sources.
- `${XDR_BASE}/runtime/state/*.json` scenario / mirror / NAT snapshots.
- `${XDR_BASE}/logs/*.jsonl` append-only logs.
- `config/caldera-lab.json` and secrets on disk (operator-owned).

**Removed or recreated on `lab destroy` / `lab cleanup`:**

- Runtime qcow2 for destroyed domains.
- Domain XML in libvirt for those names.

**Intentionally persistent (operator expectations):**

- CALDERA server state (external to this repo).
- Internal **snapshot chains** inside qcow2 until explicitly deleted with
  `lab snapshot delete <name>`.

## 4. Runtime retention guidance

### 4.1 `logs/`

- Primary structured log: `logs/caldera-orchestration.jsonl` (JSONL).
- **Growth**: proportional to scenario frequency and verbosity; high churn
  labs should **rotate** or archive monthly.

Suggested archival:

```bash
# Example: monthly copy-out (run from PROJECT_ROOT or /opt/xdr-lab)
ts="$(date -u +%Y%m%dT%H%MZ)"
mkdir -p ~/xdr-lab-log-archive/"$ts"
cp -a logs/caldera-orchestration.jsonl ~/xdr-lab-log-archive/"$ts"/
: > logs/caldera-orchestration.jsonl   # truncate only if your SIEM already ingested the copy
```

Use **copy then truncate** (or `logrotate`) — never delete the only copy
while investigating incidents.

### 4.2 `runtime/state/`

| File | Role | Retention note |
| --- | --- | --- |
| `scenario.json` | Last run, telemetry review fields | Small; keep for audit |
| `caldera.json` | Reachability / operation echo | Small |
| `mirror.json` | Mirror intent vs reality | Regenerated by mirror verbs |
| `nat.json` | Last `nat verify` snapshot | Small |

### 4.3 Snapshot catalogs

- CLI `lab snapshot list` reads libvirt plus sidecar state; large snapshot
  trees increase **qcow2** size more than JSON metadata.
- Prune old snapshot names when disk pressure mounts (after guest quiesce).

### 4.4 JSONL growth expectations

- Each **dry-run** and **live** scenario emits one or more JSONL records.
- Order-of-magnitude: **kilobytes per dry run**, **kilobytes to low megabytes
  per campaign day** in busy labs — highly variable.

### 4.5 Suggested archival strategy

1. **Daily**: monitor `df -h` on `${LIBVIRT_IMAGE_DIR}` and `${XDR_BASE}`.
2. **Weekly**: copy `logs/*.jsonl` and `runtime/state/*.json` to secure
   storage if compliance requires it.
3. **Per campaign**: export CALDERA operation UUID + `scenario.json`
   `last_live_run` block into your ticket system.
